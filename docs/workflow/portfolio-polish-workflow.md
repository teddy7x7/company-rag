# Portfolio Polish Workflow
> 適用工具：Claude Code / Cursor / 任何 Coding Agent  
> 對話模式：多輪對話，每個階段先確認大綱再展開完整內容  
> 目標：從現有 repo 快速打磨成可求職狀態，之後視需要疊代強化

---

## 使用說明

### 分級說明
- **L1（通用 MVP）**：最小可行性版本，目標是: 讓**資深技術面試官**、**科技業專業獵頭** 到 github 閱讀當前項目時，可以獲得他們的青睞，讓項目的開發者(我)「獲得面試機會」。修復致命傷、補齊基礎文檔。
- **L2（Harness 強化）**：在 L1 基礎上疊加，目標是:「項目的開發者(我)能更加脫穎而出」。強化項目  AI Harness Engineering 相關技術(如: AI 評估框架)，展現工程素養。

### 執行原則
1. 每個階段先輸出大綱，等我確認後再展開完整內容
2. 若 AI 判斷致命傷數量為 0，**階段 2 可直接跳過**
3. L2 永遠在所有 L1 完成後才執行，不要同時進行
4. 每個階段完成後，將產出物 commit 一次，保持歷史清晰

### 執行順序
```
階段 0 → 階段 1 → [階段 2，如有必要] → 階段 3 L1
→ 階段 4 L1 → 階段 5 → 階段 6
→ [若時間允許] 階段 2 L2 → 階段 3 L2 → 階段 4 L2 → 階段 5（補充）→ 階段 6（補充）
```

> **你需要提供**：目標職缺的 JD（貼在 prompt 末尾）

```

【確認清單輸出格式】

**目標職缺 JD**（確認你已收到，並列出 JD 中你認為與這個專案最相關的 3 個技術要求）

等我確認以上資訊正確且完整後，輸出到 `docs/refine/stage0-build-context.md`，再進入階段 1。
```

## 階段 1：大健檢與分級判定

> **目標**：找出致命傷、輸出分級判定，決定後續走 L1 還是 L2。  
> **前置條件**：階段 0 確認完畢

```
現在開始執行大健檢。

請扮演資深軟體架構師，結合對 JD 的理解、上一階段輸出的分析 `docs/refine/stage0-build-context.md`，繼續分析這個 repo。

【第一步：輸出大綱】
先只輸出以下大綱，不展開細節，等我確認後再逐項展開：

1. 核心業務問題
   - 這個系統解決了誰的什麼痛點？

2. 關鍵架構決策清單（只列標題，每項一行）

3. 致命傷清單（會導致面試一秒刷掉的問題，例如：密碼明碼儲存、無任何錯誤處理、嚴重記憶體洩漏）
   - 若無致命傷，明確寫出「無致命傷」

4. 一般優化建議清單（致命傷以外，只列標題）

5. 分級判定
   - 致命傷數量：
   - 架構複雜度：低 / 中 / 高
   - 與 JD 匹配程度：低 / 中 / 高
   - Harness / 評估框架現有程度：無 / 基礎 / 完整
   - 建議路線：

     □ L1 完成後，空閒時疊代 L2
     □ 建議直接做 L2（說明理由）

【推理標記規範】

- [假設]：你推測但無法從程式碼直接確認的地方
- [決策點]：你看到明顯的設計選擇
- [挑戰]：你識別出的技術難點
- [致命]：必須修復的嚴重問題

【第二步：輸出分析結果並保存】
等我確認以上資訊正確且完整後，輸出到 `docs/refine/stage1-analysis.md`，再進入階段 2。
```

## 階段 2：修復

### L1：致命傷修復

> **前置條件**：階段 1 確認有致命傷  
> **跳過條件**：階段 1 判定無致命傷，直接進階段 3

```
根據階段 1 識別出的致命傷清單，進行修復。

【第一步：輸出修復計畫大綱】
針對每一個致命傷，列出：
- 問題編號與名稱
- 問題本質（一句話說明為何嚴重）
- 修復方向（一句話）
- 預估影響範圍（哪些檔案需要改動）

只列大綱，等我確認後再輸出程式碼。

【第二步：實現修復】
將修復相關紀錄同時寫入 `docs/refine/stage2-refine-records.md`
如果問題複雜，必須先制定計畫，等我確認，再進行代碼實際修復。

【輸出程式碼時的格式要求】
每個致命傷的修復輸出：
1. 問題說明（為何嚴重）
2. 修復前程式碼（標註檔案路徑與行號）
3. 修復後程式碼
4. 一句話說明修復邏輯

注意：此階段只處理致命傷，一般優化建議暫時跳過。
```

### L2：Harness 評估框架補強

> **前置條件**：L1 所有修復完成並 commit  
> **適用情境**：有空閒時間，或目標職缺明確要求評估工程能力

```
現在針對 Harness Engineering 進行強化。

【背景】
業界對 AI 工程師的期望不只是「能讓模型跑起來」，而是「能設計可量化、可自動化驗證的評估體系」。
目標是讓這個 repo 展現出「AI 品質工程」的思維，而非只是 AI 應用開發。

【第一步：輸出評估框架缺口分析大綱】
1. 目前評估/測試現況
   - 有哪些評估機制（若有）
   - 缺少哪些業界標準元件

2. 建議補齊的評估元件清單
   每項包含：
   - 元件名稱
   - 解決什麼問題（對應到 AI 系統哪個風險）
   - 建議工具或做法（如 Ragas、ROUGE、LLM-as-a-judge、pytest + fixture）
   - 實作難度：低 / 中 / 高
   - 對求職加分程度：低 / 中 / 高

3. 建議補強優先順序（依「加分大 × 難度低」排列）

等我確認大綱後，再依序輸出實作程式碼。
如果問題複雜，必須先制定計畫，等我確認，再進行代碼實際調整。

將優化相關紀錄同時寫入 `docs/refine/stage2-refine-records.md`

【程式碼輸出格式要求】
每個評估元件：
1. 說明這個元件測試什麼、為何重要
2. 完整可執行的程式碼（含測試資料範例）
3. 如何在 README 中說明這個評估的意義
```

### 階段 2.5：疊代審查與優化儲備 (Continuous Review)
**目標**：透過「人工檢查 + AI 交叉驗證」，找出修復後引入的新問題或遺漏盲點。本階段重在「識別與紀錄」，為後續疊代建立儲備庫，暫不立即施行修復。
**前置條件**：階段 2 的 L1 致命傷修復已完成並 Commit。

【第一步：聯合審查與衍生分析 (Review & Analysis)】
請扮演嚴格的資深架構師，與我（人工）共同審視當前代碼，針對以下維度進行「二次大健檢」：

1. **修復副作用**：階段 2 的修復是否有打破原有的業務邏輯或引入新 Bug？
2. **遺漏盲點**：是否有第一階段大健檢時未被歸類為致命傷，但實際上會嚴重侵蝕系統品質的隱性問題？
3. **架構地圖落實與更新**：
   請嚴格參考 `docs/prompt/script-analysis.md` 的四個大段落（A-D）結構，針對每個重要腳本進行全局分析。
   - **思維收斂要求**：在填寫 B 段與 D 段時，必須將第 1、2 點發現的「副作用與盲點」，精準收斂並揉進對應模組（M1, M2...）的【工程實踐/效能瓶頸】與【真實場景痛點】中。
   - **輸出要求**：將這份具備全局座標系的分析結果，輸出（或更新）到 `docs/architecture` 的對應腳本分析檔案。

【第二步：輸出優化儲備計畫 (Refine Backlog)】
完成審查後，不直接修改代碼，而是將上一步發現的問題(記錄在 `docs/refine/stage2-refine-records.md`, `docs/architecture`,`docs/.private/human-review-update-draft.md`)，轉化為「技術債/優化清單」，並輸出至 `docs/refine/stage2.5-backlog.md`：

**📄 stage2.5-backlog.md 輸出格式：**
- **[問題編號] 名稱**：
- **發現管道**：[AI 識別 / 人工發現 / 聯檢碰撞]
- **嚴重程度**：高 (未來致命) / 中 (一般優化) / 低 (程式碼微調)
- **影響分析**：一句話說明放任不理的後果。
- **預計 Refine 方案**：簡述未來打算如何優化（先留紀錄，作為後續疊代依據）。

【第三步：決定進入下一階段的策略】
評估該 Backlog 清單：
- 若存在「高（未來致命）」問題，且時間允許 ➡️ 視為新 Context，重跑階段 2 修復。並且在修復完成後，根據修復過程與結果，同步更新 `stage2.5-backlog_c*.md`、`stage2-refine-records.md`、`README.md`。
- 若多為中/低度優化項目 ➡️ 封存紀錄，直接挺進「階段 3：文檔工程」，並在文檔中將此清單轉化為「未來規劃（展現成長思維）」。

【第四步：檔案有序遞增規範】
為了保持 Review 歷史清晰且不覆蓋舊紀錄，請遵循以下命名規則：
- 請檢查 `docs/refine/` 目錄下是否已存在 `stage2.5-backlog_c*.md`。
- 若不存在，本次輸出檔名為：`stage2.5-backlog_c1.md`。
- 若已存在（例如已有 c1, c2），則本次輸出請將數字自動加 1（例如：`stage2.5-backlog_c3.md`）。
- 每次重新執行本階段，請在檔案頂部標註【Review 日期】與【本次審查核心範圍】，確保可追溯性。

> **前置條件**：程式碼層面的修復已完成


## 階段 5：兵棋推演

> **前置條件**：面試官問題清單確認完畢 且 覺得 `docs/.private/stage4-interview-prep.md` 的內容某些部份需要展開深入討論時使用，否則直接跳到下一階段。  
> **注意**：此階段有兩種模式，擇一或合併使用

```
現在切換角色：你扮演一位技術熟練、對這個專案極度熟悉的求職者（我）。

這位求職者的特徵：
- 能用工程化語言解釋技術決策，不說模糊的「我覺得這樣比較好」
- 對自己系統的侷限性非常清楚，不迴避缺點，但能說明改進方向
- 用詞成熟，不會說「我有做一個 AI 然後...」

【模式 A：示範回答】
針對階段 4 的每一個問題，先輸出回答大綱（關鍵論點，3 個 bullet），
等我確認後再展開完整回答。

完整回答格式：
- 使用 STAR 原則（Situation / Task / Action / Result）
- 包含具體技術細節（框架名稱、設計模式、數據）
- 若缺乏真實數據，在句末標註【建議補充：XXX 類型的數據】
- 回答長度：口語版約 2～3 分鐘的說話量

【模式 B：批改你的回答（可選）】
你也可以先自己回答，然後用以下格式請 AI 批改：

「以下是我對問題 [N] 的回答：
[你的回答]
請從技術深度、表達清晰度、STAR 完整性三個維度評分（1～10），
並給出具體改進建議。」
```

## 附錄：常見問題

**Q：我的 repo 還沒有任何測試，還適合用這個流程嗎？**  
A：適合。階段 1 會識別這個缺口，L1 不強制要求測試，L2 會幫你補齊最有說服力的評估腳本。

**Q：JD 要不要在每個階段都重新貼？**  
A：同一個對話中不需要，AI 會記得。開新對話時，在階段 0 重新提供即可。

**Q：可以同時針對多個 JD 優化嗎？**  
A：建議每次只針對一個目標 JD，保持文檔的說服力集中。不同 JD 開新對話重跑。
