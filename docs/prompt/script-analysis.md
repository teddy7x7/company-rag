# Script Analysis Prompt Template

## 全局分析模式（首次分析時使用）

請完整分析下面這個 python 腳本 <script-name>.py,這是一次性的全局分析,
之後我們會基於這份分析逐模塊精讀,所以請用以下固定結構整理,
讓它可以作為後續所有對話的「大綱文件」被反覆引用:

## A. 整體定位
- 這個檔案在整個系統/專案中扮演什麼角色?依賴哪些外部模組?
- 小結: 用一句話講「這個腳本存在的理由」(它解決什麼問題)

## B. 模塊劃分與建議閱讀順序
請把整個腳本切成 3-6 個「主要模塊」(依語意區塊切,例如:
資料結構定義、核心演算法、邊界處理、輸出呈現等,不要機械地按行數切)。
每個模塊請給:
- 模塊編號與名稱(例如:M1. 資料結構設計)
- 對應的行號範圍
- 一句話概要(這個模塊在幹什麼)
- 預估閱讀難度(入門/進階/硬核) 與 技術稀缺度(以 AI engineer 角度)
- 核心資料結構
- 關鍵知識點
- 工程實踐 / 效能瓶頸 (簡短描述)

切完之後,請額外給出「建議精讀順序」——這不一定等於模塊在檔案裡
從上到下的順序,而是一個資深工程師第一次拿到這份代碼時實際會選擇的
理解順序(例如:可能會先讀含入口函式/主流程的模塊抓主線,
再回頭看它依賴的資料結構定義,即使那段定義寫在檔案更前面)。
每個順序請附一句理由。

## C. 整體流程圖(巨觀地圖)
請用 Mermaid 語法畫出整體執行流程,要求:
- 圖中每個主要節點/子圖,必須直接對應 B 段切出的模塊編號(M1, M2, M3...),
  不要另外發明一套節點命名,確保這張圖跟模塊表是同一套座標系統
- 標示出模塊之間的資料流向與關鍵判斷分支
- 這張圖只需要巨觀層級,不需要展開模塊內部的細節邏輯
  (內部細節會在我們實際精讀到該模塊時,另外畫局部放大圖,見 Stage 1)

函式呼叫關係圖**不是預設要畫**,只有當呼叫關係符合以下任一條件時才額外畫:
存在遞迴呼叫、同一函式被多個不同模塊呼叫、或呼叫鏈深度超過 3 層。
如果不符合,只需要用一句話描述呼叫關係即可。

## D. 商業場景落地與工程價值 (Business Value & Resume Highlights)
請將本架構的設計核心，包裝成具備商業洞察與工程價值的論述：

1. 真實場景痛點與解決方案
- **情境帶入**：描述在真實業務或 Production 環境中，若缺乏此系統，工程師會面臨的 2-3 個「棘手痛點」（如：優化盲目、人工測試成本高昂、架構高度耦合難以追蹤等）。
- **對症下藥**：針對每一個痛點，**必須具體引用前面定義的模塊編號（M1, M2...）**，解釋本架構是如何透過這些模塊提供客觀定量、自動化或解耦的解決方案。

2. 核心技術亮點 (Key Highlights)
- 提煉 2-3 點高含金量的工程實踐亮點（例如：端到端自動化管線的建立、嚴謹的資料合約約束、非同步/串流機制的預留等），使用高級軟體工程術語撰寫。

## 增量更新模式（代碼修復後使用）

當已存在該腳本的全局分析文件（`docs/architecture/*-overview.md`），
且代碼已發生修改時（如 Backlog 修復後），請使用以下精簡流程：

1. **先閱讀**既有的 `*-overview.md`，確認目前的模塊編號體系（M1, M2...）
2. **只更新**受影響的段落，保持模塊編號不變（除非新增了全新的模塊）
3. **更新紀錄**：在文件頂部標註 `> 最後更新：{date} — 因 {B-XX 修復} 更新 {段落}`
4. **不要重寫整份文件**，只做 diff-level 的精準更新

### 觸發條件
- Stage 2.5 循環（c2, c3...）時
- Backlog 項目被修復後（特別是影響模塊結構的修復）
- 新增重要功能（如 B-08 Guardrails 會新增 M-new 模塊）
